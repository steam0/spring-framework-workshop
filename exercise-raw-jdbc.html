Raw JDBC

In this exercise you'll build a user API step by step — starting simple, then swapping implementations to see why interfaces and dependency injection matter.

Setup

  1. Go to start.spring.io
  2. Configure: Spring Boot 4.0.x, Kotlin, Maven
  3. Add only Spring Web as a dependency
  4. Download and open the project in your IDE

Step 1 — RestController

Create a @RestController with a GET /users/{id} endpoint that returns a single user. Each user has:

  • userId: Long
  • name: String

For now, just return a hardcoded user. Verify it works: curl http://localhost:8080/users/1

Step 2 — UserRepository interface

Create a UserRepository interface with a single method:

interface UserRepository {
    fun fetchUser(userId: Long): User?
}

Update your controller to depend on this interface instead of hardcoding the data.

Step 3 — HashMap implementation

Create a class InMemoryUserRepository that implements UserRepository. Store users in a HashMap<Long, User> and look up by ID in fetchUser().

Register it as a @Component and verify your endpoint still works.

Step 4 — H2 implementation

Now create a second implementation: JdbcUserRepository. This one connects to an actual database.

  1. Add H2 as a dependency in your pom.xml: 
    <dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
</dependency>
  2. Create a @Bean that returns an H2 DataSource
  3. In JdbcUserRepository, inject the DataSource, get a connection, and run SQL. Create the table and seed it in a @PostConstructA standard Java/Jakarta annotation. Spring calls this method automatically after the bean is created and all dependencies are injected — but before the application starts serving requests. Useful for one-time setup like creating tables.: 
    CREATE TABLE IF NOT EXISTS users (
    user_id BIGINT PRIMARY KEY,
    name VARCHAR(255)
);

INSERT INTO users VALUES (1, 'Ola Nordmann');
INSERT INTO users VALUES (2, 'Kari Nordmann');
INSERT INTO users VALUES (3, 'Per Hansen');
    Then implement fetchUser() with a SELECT by ID.
  4. Switch from the HashMap implementation to the JDBC one

Step 5 — NamedParameterJdbcTemplate

Raw JDBC works, but it's painful — manual connection handling, positional ? parameters, and you have to remember to close everything. Add spring-boot-starter-jdbc and create a third implementation using NamedParameterJdbcTemplate:

  1. Add the dependency: 
    <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jdbc</artifactId>
</dependency>
  2. Add application.yml with the H2 datasource URL — Spring Boot auto-configures the DataSource and NamedParameterJdbcTemplate for you
  3. Create a NamedJdbcUserRepository that implements UserRepository
  4. Inject NamedParameterJdbcTemplate and use named parameters (:userId) instead of ?
  5. Mark it @Primary so it replaces the raw JDBC version

Compare: no manual connection handling, no close(), named parameters instead of positional ones. This is what we use in production.

Hints

H2 in-memory URL
Use jdbc:h2:mem:testdb — the database lives in memory and disappears when the process stops.
H2's built-in DataSource
H2 ships with org.h2.jdbcx.JdbcDataSource — you can create one directly without Spring.
Two implementations, one interface
You have two @Component classes implementing UserRepository. Use @Primary or @Qualifier to pick which one Spring injects.
The point of the exercise
Because the controller depends on the interface, swapping from HashMap to H2 required zero changes in the controller. That's the power of DI.
Notice how painful raw JDBC is — manual connection handling, SQL strings, ResultSet iteration. Later, Spring Data JPA will eliminate all of this.
