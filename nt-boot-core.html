nt-boot-core

This module contains what every NT application must have. It exists because our Container Pipeline has strict requirements for all applications — nt-boot-core auto-configures them so you don't have to.

Pipeline requirement → nt-boot-core solution

Structured JSON logging
Logstash encoder on by default — every log line is JSON to stdout
Application metrics (OpenMetrics)
Prometheus endpoint on /actuator/prometheus
Health probes for Kubernetes
Liveness, readiness, and startup probes pre-configured
Traceability through environments
application name and deployment_number in every log line and metric

NtApplicationConfig

A @ConfigurationProperties bean bound to the nt.application prefix. It gives every module and every log line a consistent identity:

clientId
Your Maven artifactId
<groupId>no.norsktipping</groupId>
<artifactId>my-service</artifactId>
<version>1.0.0</version>
The artifactId is the name of your project as declared in pom.xml. Spring Boot's build-info plugin makes it available at runtime via BuildProperties.
(from BuildProperties)
clientVersion
Deployment numberAn auto-generated number assigned by the Container Pipeline during the build. It gives full traceability from commit to production — every log line, metric, and container image can be traced back to the exact build.
(from DEPLOYMENT_NUMBER env)
vendorId
Defaults to
"Norsk-Tipping"

You never need to set these manually — they're auto-populated from BuildProperties and the pipeline environment. Other nt-boot modules (Unleash, Kafka, Feign) inject NtApplicationConfig to identify themselves consistently.

Structured JSON Logging

On by default. Every log statement automatically includes clientId and clientVersion from NtApplicationConfig, giving full traceability from commit to production.

For local development, switch to human-readable output:

LOG_APPENDER=CONSOLE
